## Problem Statement
1. 当前 `quality_grade` 既表示“验证时的基础质量”，又被 `session.adjustGrade()` 当成“运行时风险惩罚”的持久化状态，语义已经混杂。
2. 代理选择、替换、健康检查、会话过滤都以这个单字段为依据，所以只改一个函数不够，必须处理消费链路。
3. 只按 `latency` 打分会把“偶尔很快但经常失败”的代理误判为高质量，也无法体现使用成功率和风险记录。

## Proposed Design
### 1. 数据模型
- 在 `proxies` 表新增 `quality_score INTEGER NOT NULL DEFAULT 0`。
- 保留 `quality_grade TEXT` 作为外部兼容字段，由 `quality_score` 映射得出。
- 新增最小惩罚字段，优先考虑 `risk_count INTEGER NOT NULL DEFAULT 0`。
  - 原因：当前会话层只有“降档”副作用，没有可重算的原始风险信号。
  - 目标：把风险记录沉淀为输入，让等级可重算，而不是直接改坏最终等级。

### 2. 评分模型
- 引入统一评分函数，建议放在 `storage` 包内单独文件，例如 `storage/quality.go`。
- 第一版采用 0-100 分整数分，权重先固定在代码中，不额外开放配置项：
  - latency: 50-60 分
  - 成功率/失败率: 20-25 分
  - 使用稳定性: 10-15 分
  - 风险惩罚: 10-15 分
- 评分输入优先复用现有字段：
  - `latency`
  - `success_count`
  - `fail_count`
  - `use_count`
  - `risk_count`
- `quality_grade` 由 score 分段映射：
  - 建议首版采用 `S >= 85`, `A >= 70`, `B >= 50`, `C < 50`
  - 最终阈值在实现时按样本分布微调，但要写成常量，避免散落各处。

### 3. 写入链路改造
- `UpdateExitInfo()` 改为：
  - 更新出口信息、延迟
  - 读取当前统计字段
  - 统一计算 `quality_score` 与 `quality_grade`
- `ReplaceProxy()` 插入新代理时同时写入初始 score/grade。
- 会话释放时不再直接把 `quality_grade` 降档写死。
  - 改为更新 `risk_count`，然后通过统一评分函数重算 `quality_score` 和 `quality_grade`。

### 4. 消费链路改造
- 存储排序：
  - `GetRandom()`、`GetAll()`、`GetBatchForHealthCheck()`、`GetWorstProxies()` 优先改为以 `quality_score` 为主排序，`latency` 为次级排序。
- 池子替换：
  - `pool.Manager.tryReplace()` 先比较综合质量，再在同级时比较延迟。
  - 目标是避免“低延迟但不稳定”的新代理把“略慢但更稳”的旧代理换掉。
- 优化器：
  - 候选保留条件不再只看 `MaxLatencyHealthy`，而是要求最低综合等级或最低 score。
  - 可保留延迟上限作为硬门槛，防止极慢代理因统计数据少而被误选。
- 健康检查：
  - “跳过 S 级”仍保留，但其语义改为“综合最优代理”。
- 会话选择：
  - `min_grade` 继续走 `S/A/B/C`，但排序改为 `quality_score DESC, latency ASC`。

### 5. WebUI 与文档
- API 返回中新增 `quality_score`，保留 `quality_grade`。
- WebUI 至少增加一个可见分数列或详情提示，否则用户无法理解为什么“同为 A 级”的代理排序不同。
- 质量分布条仍按 `S/A/B/C` 聚合，避免界面改动过大。
- README 更新“质量分级标准”与“核心机制详解”，明确说明：
  - 等级来源于综合评分，不再是纯延迟阈值
  - 延迟仍然是核心因子，但不是唯一因子

7. 增加测试并做最小回归验证。

## Risks
- 历史 `success_count/use_count/fail_count` 数据分布可能稀疏，首版权重过激会造成大量代理掉到低等级。
- 如果继续保留 `quality_grade` 作为部分 SQL 排序键，容易出现新旧逻辑混用。
- 会话层当前直接写 `quality_grade`，如果改造不彻底，可能出现“释放后等级闪回”。
- WebUI 若不展示分数，用户会看到“等级没怎么变，但选择结果变了”，解释成本高。

## Assumptions
- 首版先做“内部综合评分”，不引入外部信誉源。
- 国家/地区信息暂时继续作为过滤条件，不直接参与评分主权重。
- 现有 `S/A/B/C` 语义需要兼容，因为 `session` API 已对外暴露 `min_grade`。
